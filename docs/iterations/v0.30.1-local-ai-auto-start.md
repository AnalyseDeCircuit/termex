# v0.30.1 — Local AI Auto-Start: 本地 AI 引擎自动启动

> 状态：**计划中**
> 依赖：v0.30.0
> 关联：v0.5.0 (AI Core)、v0.11.0 (本地 AI 模型)

---

## 背景

### 当前问题

1. **用户不知道本地引擎未启动** — 设置中配置了默认 Local AI provider，但引擎实际未运行。智能补全和 AI 对话静默失败，返回空结果，用户完全无感知
2. **每次启动都要手动选模型** — 打开 AI 面板 → 看到"Local engine is not running" → 选模型 → 点 Start，操作路径太长
3. **多窗口场景** — 用户可能打开多个 Termex 窗口，每个窗口都尝试启动 llama-server 会端口冲突
4. **退出时残留** — 如果不自动关闭，llama-server 进程可能在所有窗口关闭后继续占用资源

### 目标

1. 应用启动后自动恢复上一次使用的本地模型，减少手动操作
2. AI 面板提供清晰的启动状态提示 + 模型切换入口
3. 多窗口共享同一个 llama-server 实例，避免重复启动
4. 最后一个窗口关闭时自动停止引擎

## 一、自动启动策略

### 1.1 启动流程

```
应用启动 (lib.rs setup)
  │
  ├─ 读取 settings: "local_ai_last_model" (上次使用的模型 ID)
  │
  ├─ 如果没有保存的模型 → 不自动启动，走现有逻辑
  │
  ├─ 如果有保存的模型:
  │   ├─ 检查该模型文件是否存在 → 不存在则跳过
  │   ├─ 检查端口 15000 是否已被占用（另一个窗口已启动）
  │   │   ├─ 已占用 → 复用：设置 state.llama_server.port = 15000，不启动新进程
  │   │   └─ 未占用 → 延迟 3 秒后自动启动
  │   │       └─ 前端发送通知："3 秒后启动本地 AI (qwen2.5-0.5b-q3)... [取消]"
  │   └─ 启动成功 → 更新状态 + 前端通知
  │
  └─ 前端 AI 面板自动弹出启动提示
```

### 1.2 Settings KV

| Key | Value 示例 | 说明 |
|-----|-----------|------|
| `local_ai_last_model` | `"qwen2.5-0.5b-q3"` | 上次启动的模型 ID |
| `local_ai_auto_start` | `"true"` | 是否自动启动（用户可关闭） |

### 1.3 后端实现

**修改文件**: `src-tauri/src/lib.rs` — setup 回调中新增自动启动逻辑

```rust
// 在 setup() 中，settingsStore.loadAll() 完成后:
{
    let state = app.state::<AppState>();
    let app_handle = app.handle().clone();

    tokio::spawn(async move {
        // 读取上次使用的模型
        let last_model = state.db.with_conn(|conn| {
            crate::storage::settings::get(conn, "local_ai_last_model")
        }).ok().flatten();

        let auto_start = state.db.with_conn(|conn| {
            crate::storage::settings::get(conn, "local_ai_auto_start")
        }).ok().flatten().unwrap_or_else(|| "true".to_string());

        if auto_start != "true" || last_model.is_none() {
            return;
        }
        let model_id = last_model.unwrap();

        // 检查模型文件是否存在
        let model_path = crate::paths::models_dir().join(format!("{model_id}.gguf"));
        if !model_path.exists() {
            return;
        }

        // 检查是否有其他窗口已启动 llama-server（端口探测）
        if is_port_in_use(15000).await {
            log::info!("[local-ai] port 15000 occupied, reusing existing instance");
            let mut server = state.llama_server.write().await;
            server.port = Some(15000);
            let _ = app_handle.emit("local-ai://status", serde_json::json!({
                "running": true,
                "model": model_id,
                "reused": true,
            }));
            return;
        }

        // 延迟 3 秒，给前端时间展示取消提示
        let _ = app_handle.emit("local-ai://auto-start", serde_json::json!({
            "model": model_id,
            "delayMs": 3000,
        }));

        tokio::time::sleep(std::time::Duration::from_secs(3)).await;

        // 检查用户是否取消了（通过 state flag）
        if state.local_ai_auto_start_cancelled.load(std::sync::atomic::Ordering::Relaxed) {
            return;
        }

        // 启动
        match crate::local_ai::start_server(&state, &app_handle, &model_id).await {
            Ok(_) => {
                log::info!("[local-ai] auto-started model: {model_id}");
                let _ = app_handle.emit("local-ai://status", serde_json::json!({
                    "running": true,
                    "model": model_id,
                    "reused": false,
                }));
            }
            Err(e) => {
                log::warn!("[local-ai] auto-start failed: {e}");
                let _ = app_handle.emit("local-ai://auto-start-failed", serde_json::json!({
                    "model": model_id,
                    "error": e.to_string(),
                }));
            }
        }
    });
}
```

## 二、多窗口共享

### 2.1 端口探测

```rust
async fn is_port_in_use(port: u16) -> bool {
    // 方式一：尝试 TCP connect
    tokio::net::TcpStream::connect(("127.0.0.1", port))
        .await
        .is_ok()
}
```

### 2.2 共享逻辑

| 场景 | 行为 |
|------|------|
| 窗口 A 启动 llama-server | 正常启动，占用端口 15000 |
| 窗口 B 启动 | 检测到 15000 已占用 → 复用，不启动新进程 |
| 窗口 B 关闭 | 不停止 llama-server（窗口 A 还在用） |
| 窗口 A 关闭（最后一个窗口） | 停止 llama-server 进程 |
| 窗口 B 手动切换模型 | 停止旧进程 → 启动新进程（窗口 A 也会使用新模型） |

### 2.3 最后窗口关闭检测

**修改文件**: `src-tauri/src/lib.rs` — window destroy handler

```rust
// 在现有 window destroy handler 中:
// 检查是否是最后一个窗口
let remaining_windows = app.webview_windows().len();
if remaining_windows <= 1 {
    // 最后一个窗口关闭 — 停止 llama-server
    let state = app.state::<AppState>();
    let server = state.llama_server.read().await;
    if let Some(ref child) = server.process {
        let _ = child.kill();
    }
}
```

### 2.4 复用检测 vs 进程所有权

引入 `is_owner` 标记区分"自己启动的"和"复用的"：

```rust
pub struct LlamaServerState {
    pub port: Option<u16>,
    pub process: Option<Child>,
    pub model_id: Option<String>,
    /// true = this instance started the process; false = reusing another window's process
    pub is_owner: bool,
}
```

只有 `is_owner = true` 的窗口才在关闭时 kill 进程。

## 补充设计 A：model_id 与 model_path 映射

### A.1 问题

当前 `local_ai_start_engine` 接收完整文件路径（`model_path: String`），不是 model_id。设计文档用 `model_id`（如 `qwen2.5-0.5b-q3`）做 last_model 持久化，但启动时需要完整路径。

### A.2 方案

`local_ai_last_model` 存储 **model_id**（文件名去 `.gguf` 后缀），启动时从 `models_dir()` 拼接路径：

```rust
// 保存 (local_ai_start_engine 成功后):

    .file_stem()
    .map(|s| s.to_string_lossy().to_string())
    .unwrap_or_default();
set_setting(state, "local_ai_last_model", &model_id);

// 恢复 (自动启动时):
let model_id = get_setting(state, "local_ai_last_model")?;
let model_path = crate::paths::models_dir().join(format!("{model_id}.gguf"));
if !model_path.exists() {
    log::info!("[local-ai] model file not found: {}", model_path.display());
    return; // 不自动启动
}
let model_path_str = model_path.to_string_lossy().to_string();
// 调用现有的 start() 函数
```

### B.1 问题

设计文档用 `child.kill()` 和 `process: Option<Child>`，但实际代码通过 PID 管理进程（`process_id: Option<u32>`），使用 `kill -9` 停止。

### B.2 修正

```rust
pub struct LlamaServerState {

    pub port: Option<u16>,

impl LlamaServerState {
    pub fn new() -> Self {
        Self {
            process_id: None,
            port: None,
            loaded_model: None,
            is_owner: false,

}
```

在 `start()` 成功后设 `is_owner = true`。在端口复用时设 `is_owner = false`。

## 补充设计 D：Orphan Cleanup 顺序修正

### D.1 问题

当前 `kill_orphaned_llama_servers()` 在启动时无差别杀掉所有 llama-server 进程。在多窗口场景下，窗口 B 启动时会误杀窗口 A 启动的实例。

### D.2 修正后的启动顺序

```
应用启动
  │
  ├─ 1. 尝试端口探测 (try_recover_llama_state)
  │   ├─ 发现运行实例 → 复用 (is_owner=false)，跳到步骤 4
  │   └─ 未发现 → 继续
  │
  ├─ 2. 读取 PID 文件 (~/.termex/.llama-server.pid)
  │   ├─ PID 存在且进程存活 → 复用，跳到步骤 4
  │   ├─ PID 存在但进程已死 → 清理 PID 文件，继续
  │   └─ 不存在 → 继续
  │
  ├─ 3. 清理 orphan（仅在步骤 1&2 都未发现时执行）
  │   └─ kill_orphaned_llama_servers()
  │
  ├─ 4. 自动启动（如果有 last_model 且未复用）
  │   └─ 延迟 3 秒 → 启动 → 写入 PID 文件
  │
  └─ 5. 通知前端
```

### D.3 PID 文件管理

```rust
const PID_FILE: &str = ".llama-server.pid";

/// 写入 PID 文件
fn write_pid_file(pid: u32, port: u16) -> std::io::Result<()> {
    let path = crate::paths::data_dir().join(PID_FILE);
    std::fs::write(&path, format!("{pid}\n{port}"))?;
    Ok(())
}

/// 读取 PID 文件
fn read_pid_file() -> Option<(u32, u16)> {
    let path = crate::paths::data_dir().join(PID_FILE);
    let content = std::fs::read_to_string(&path).ok()?;
    let mut lines = content.lines();
    let pid: u32 = lines.next()?.parse().ok()?;
    let port: u16 = lines.next()?.parse().ok()?;
    Some((pid, port))
}

/// 删除 PID 文件
fn remove_pid_file() {
    let path = crate::paths::data_dir().join(PID_FILE);
    let _ = std::fs::remove_file(&path);
}

/// 检查 PID 对应的进程是否存活
fn is_process_alive(pid: u32) -> bool {
    #[cfg(unix)]
    {
        unsafe { libc::kill(pid as i32, 0) == 0 }
    }
    #[cfg(windows)]
    {
        // tasklist /FI "PID eq {pid}" 检查
        std::process::Command::new("tasklist")
            .args(&["/FI", &format!("PID eq {pid}"), "/NH"])
            .output()
            .map(|o| String::from_utf8_lossy(&o.stdout).contains(&pid.to_string()))
            .unwrap_or(false)
    }
}
```

## 补充设计 F：取消自动启动的时序处理

### F.1 问题

用户可能在 3 秒倒计时结束后、进程启动过程中（可能需要数秒加载模型）才点取消。此时进程已在启动，`AtomicBool` 检查已过。

### F.2 方案

两阶段取消：

```rust
// 阶段 1: 3 秒延迟期间检查
tokio::time::sleep(Duration::from_secs(3)).await;
if state.local_ai_auto_start_cancelled.load(Ordering::Relaxed) {
    return; // 延迟期间取消 — 不启动
}

// 阶段 2: 启动过程中也检查
// 修改 start() 函数，接受 CancellationToken 参数
let cancel_token = state.local_ai_auto_start_cancelled.clone();
match start_with_cancellation(&state, &app_handle, &model_id, &cancel_token).await {
    Ok(_) => { /* 成功 */ }
    Err(e) if e.to_string().contains("cancelled") => {
        log::info!("[local-ai] auto-start cancelled during startup");
        // 如果已经启动了进程，停止它
        let mut server = state.llama_server.write().await;
        if server.is_owner {
            server.stop().await;
            remove_pid_file();
        }
    }
    Err(e) => { /* 其他错误 */ }
}
```

## 补充设计 H：最后窗口关闭检测修正

### H.1 问题

`on_window_event(Destroyed)` 触发时，检查 `webview_windows().len()` 可能不可靠（窗口已销毁）。

### H.2 方案：引用计数

在 `AppState` 中维护窗口计数：

```rust
pub struct AppState {
    // ... existing fields ...
    pub window_count: std::sync::atomic::AtomicU32,
}
```

```rust
// lib.rs setup:
app.on_window_event(|window, event| {
    match event {
        tauri::WindowEvent::Focused(true) => {
            // 窗口创建/激活时不增加计数（在 window create 事件中增加）
        }
        tauri::WindowEvent::Destroyed => {
            if let Some(state) = window.app_handle().try_state::<AppState>() {
                let prev = state.window_count.fetch_sub(1, Ordering::SeqCst);
                if prev <= 1 {
                    // 最后一个窗口 — 停止 llama-server
                    let server = state.llama_server.blocking_lock();
                    if server.is_owner {
                        // kill via PID
                        remove_pid_file();
                    }
                }
            }
        }
        _ => {}
    }
});
```

或更简单 — 依赖现有的 `Drop` 机制。`AppState` 在最后一个窗口销毁后被 drop，`LlamaServerState::drop()` 自动触发 kill（仅 `is_owner = true` 时）。

**推荐方案**：保持现有 `Drop` 机制 + 加入 `is_owner` 判断。不需要窗口计数。
