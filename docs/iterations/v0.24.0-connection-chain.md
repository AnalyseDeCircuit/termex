# V0.24.0 — Connection Chain Refactor (Flexible Target Position + Exit Routing)

## 背景

Termex 的 SSH 连接路径一直是固定的：`本地 → [Network Proxy] → [SSH Bastion] → Target`。Target 始终是链路的最后一个节点，也是最终的出口节点。这意味着从 Target 访问任何外部服务时，可见 IP 地址永远是 Target 的 IP。

**用户需求**：将 Target 放在连接链路的任意位置，使得 Target 之后的节点（SSH 跳板机或网络代理）成为出口 IP。

### 目标连接路径示例

| 路径 | 出口 IP |
|------|---------|
| `Local → SSH → Proxy → Target` | Target (当前行为) |
| `Local → SSH → Target → Proxy` | Proxy |
| `Local → Target → SSH → Proxy` | Proxy |
| `Local → SSH → Proxy1 → Target → Proxy2` | Proxy2 |
| `Local → SSH1 → Proxy1 → Target → SSH2 → Proxy2` | Proxy2 |

### 当前状态

- **已实现**: 单个 SSH Bastion (`proxy_id`) + 单个 Network Proxy (`network_proxy_id`)
- **已实现**: `resolve_proxy_chain()` / `connect_via_chain()` 仅支持单跳
- **已实现**: 前端 `ChainHop` 拖拽排序 UI（但只取第一个 proxy + 第一个 bastion）
- **未实现**: 多跳链式连接、Target 可移动位置、post-target 出口路由

### 设计决策

- 新增 `connection_chain` 数据表取代 `proxy_id` / `network_proxy_id` 的单一关联模型
- 链路以 `phase` 字段区分 pre-target（到达 Target 的入口链路）和 post-target（出口路由链路）
- 旧字段保留不删除（SQLite 无法删除列），新代码优先读 `connection_chain`，回退到旧字段
- Post-target 出口路由通过 SSH Remote Port Forward + 本地 SOCKS5 代理 + 自动注入环境变量实现
- 不新增 Cargo 依赖（所有功能基于现有 `russh`, `tokio`, `tokio-socks`, `tokio_util` 实现）

## 一、数据库：Migration V10

### 1.1 新表 `connection_chain`

```sql
CREATE TABLE connection_chain (
    id          TEXT PRIMARY KEY,
    server_id   TEXT NOT NULL,         -- FK → servers.id (Target server)
    position    INTEGER NOT NULL,      -- 0-based 排序
    hop_type    TEXT NOT NULL,          -- "ssh" | "proxy"
    hop_id      TEXT NOT NULL,          -- servers.id (ssh) 或 proxies.id (proxy)
    phase       TEXT NOT NULL DEFAULT 'pre',  -- "pre" | "post"
    created_at  TEXT NOT NULL,
    FOREIGN KEY (server_id) REFERENCES servers(id) ON DELETE CASCADE
);
CREATE INDEX idx_chain_server ON connection_chain(server_id, position);
```

**关键设计**:
- Target 本身不作为 chain 行存储 —— Target 就是拥有这条 chain 的 server 记录本身
- `phase = "pre"` 表示 Target 之前的跳板（入口链路）
- `phase = "post"` 表示 Target 之后的节点（出口路由）
- 级联删除：删除 server 时自动清理其 chain 行

### 1.2 数据迁移

Migration V10 自动将所有现有 server 的 `network_proxy_id` 和 `proxy_id` 转换为 `connection_chain` 行：
- `network_proxy_id` → `{position: 0, hop_type: "proxy", phase: "pre"}`
- `proxy_id` → `{position: 1, hop_type: "ssh", phase: "pre"}`

**文件**: `src-tauri/src/storage/migrations.rs` (V10 + `migrate_legacy_chains()`)

### 1.3 Chain CRUD

**新文件**: `src-tauri/src/storage/chain.rs`

```rust
pub fn list(db, server_id) -> Vec<ChainHop>    // 按 position 排序
pub fn save(db, server_id, hops)               // DELETE + INSERT 全量替换
pub fn delete(db, server_id)                   // 清空指定 server 的 chain
```

## 二、多跳连接引擎

### 2.1 核心架构

**新文件**: `src-tauri/src/ssh/chain_connect.rs`

引入 `ChainState` 有限状态机，逐跳推进连接状态：

```
ChainState::Start
    ↓ SSH hop → connect() + auth → pool
ChainState::SshSession { server_id }
    ↓ Proxy hop → direct-tcpip to proxy host:port
ChainState::ProxyStream { stream, config }
    ↓ SSH hop → proxy handshake → SSH over stream → pool
ChainState::SshSession { server_id }
    ↓ ... (继续)
    ↓ 最终 → connect_target()
```

### 2.2 跳转组合矩阵

| 当前状态 | 下一跳 | 动作 |
|---------|--------|------|
| Start | SSH | `SshSession::connect()` → 认证 → 入池 |
| Start | Proxy | TCP connect to proxy → 存储 stream + config |
| SshSession | SSH | `connect_via_proxy()` (direct-tcpip) → 认证 → 入池 |
| SshSession | Proxy | `direct-tcpip` to proxy host:port → 存储 stream |
| ProxyStream | SSH | proxy handshake → `connect_on_stream()` → 认证 → 入池 |
| ProxyStream | Proxy | prev proxy handshake → reach new proxy → 存储 stream |

### 2.3 目标连接

最后一跳到 Target 根据当前状态：
- `Start` → 直连 `SshSession::connect()`
- `SshSession` → `connect_via_proxy()` (direct-tcpip)
- `ProxyStream` → proxy handshake + `connect_on_stream()`

### 2.4 Bastion 会话池

与现有设计一致，所有 SSH 跳板机存入 `state.proxy_sessions` 池，以 `server_id` 为键、引用计数管理生命周期。多个 Target 可共享同一 bastion 连接。

**变更**: `proxy_sessions` 类型从 `TokioRwLock<HashMap<...>>` 改为 `Arc<TokioRwLock<HashMap<...>>>`，使 exit proxy 后台任务可持有引用。

## 三、Proxy-over-SSH 隧道

### 3.1 新方法 `connect_via_proxy_on_stream()`

**文件**: `src-tauri/src/ssh/proxy.rs`

在已有 `connect_via_proxy()`（创建新 TCP 连接到 proxy）基础上，新增在**已有 stream** 上执行 proxy 握手的版本。用于"SSH 隧道内的 proxy"场景：

```rust
pub async fn connect_via_proxy_on_stream(
    proxy: &ProxyConfig,
    target_host: &str,
    target_port: u16,
    stream: Box<dyn AsyncStream>,    // 例如 SSH direct-tcpip channel
) -> Result<Box<dyn AsyncStream>, SshError>
```

实现了 SOCKS5、HTTP CONNECT、SOCKS4 三种协议的 on-stream 版本，手工实现协议握手（不依赖 `tokio-socks`，因为它需要 `TcpStream`）。

### 3.2 新方法 `connect_via_tunneled_proxy()`

**文件**: `src-tauri/src/ssh/session.rs`

组合 direct-tcpip + proxy handshake + SSH handshake 的完整链路：

```
bastion → direct-tcpip to proxy:port → proxy handshake → SSH to target
```

### 3.3 新方法 `connect_on_stream()`

**文件**: `src-tauri/src/ssh/session.rs`

通过任意 `AsyncStream` 建立 SSH 连接，供 chain engine 在各种传输层上启动 SSH 握手。

## 四、Post-Target 出口路由

### 4.1 工作原理

当 Target 后面有节点时（`phase = "post"`），Termex 自动设置出口代理：

```
Target shell → 127.0.0.1:<exit_port> (SOCKS5)
     ↓ SSH Remote Port Forward
Termex 本地 SOCKS5 server → 127.0.0.1:<local_port>
     ↓ direct-tcpip / proxy handshake
Exit SSH / Exit Proxy → Internet
```

### 4.2 SOCKS5 Exit Proxy

**新文件**: `src-tauri/src/ssh/exit_proxy.rs`

```rust
// 通过 SSH 会话池中的 exit SSH 节点路由
pub async fn start_exit_socks5_via_pool(state, exit_server_id, cancel) -> (port, task)

// 通过 Target direct-tcpip 到 proxy 路由（预留接口）
pub async fn start_exit_socks5_via_proxy_through_target(...)
```

每个 SOCKS5 CONNECT 请求通过 exit SSH 会话的 `channel_open_direct_tcpip` 路由到目标地址。

### 4.3 环境变量自动注入

在 `ssh_open_shell` 中，当 `session.exit_proxy_port` 存在时，自动向 shell 写入：

```bash
export ALL_PROXY=socks5://127.0.0.1:<port> HTTP_PROXY=socks5://127.0.0.1:<port> HTTPS_PROXY=socks5://127.0.0.1:<port> http_proxy=socks5://127.0.0.1:<port> https_proxy=socks5://127.0.0.1:<port>
```

使 `curl`, `wget`, `apt` 等工具自动通过出口链路发出请求。

### 4.4 清理

断开连接时：
1. `CancellationToken` 取消 SOCKS5 后台任务
2. 递减所有 bastion 的引用计数（pre + post 链路）
3. 关闭 SSH 会话

## 五、命令层重构

### 5.1 `ssh_connect` 重构

**文件**: `src-tauri/src/commands/ssh.rs`

原有的 4-branch 逻辑（`network_proxy` x `bastion` 的 4 种组合）替换为 chain engine 调用：

```rust
// 1. 加载 chain 数据（V10+ 优先，回退到 proxy_id/network_proxy_id）
let chain_hops = storage::chain::list(&state.db, &server_id);
let (pre_hops, post_hops) = if !chain_hops.is_empty() {
    resolve_chain_hops(&state, &chain_hops)
} else {
    build_legacy_pre_hops(&state, &server)  // 旧字段回退
};

// 2. 解析凭据
let resolved_target = resolve_target_info(&state, &server);

// 3. 通过 chain engine 连接
let result = chain_connect::connect_chain(state, &app, &status_event,
    pre_hops, resolved_target, post_hops).await;
```

### 5.2 `ServerInput` 扩展

**文件**: `src-tauri/src/commands/server.rs`

```rust
pub struct ServerInput {
    // ... 原有字段 ...
    pub chain: Vec<ChainHopInput>,  // 新增
}
```

`server_create` 和 `server_update` 在保存 server 行后调用 `storage::chain::save()` 持久化 chain。
`server_list` 加载每个 server 的 chain 数据。

### 6.2 ConnectModal 链路 UI

**文件**: `src/components/sidebar/ConnectModal.vue`

- `ChainHop` → `ChainNode`，新增 `type: "target"` 节点类型
- Target 节点始终存在（默认初始为 `[{ type: "target", id: "" }]`）
- Target 可拖拽到链路任意位置（复用现有鼠标拖拽逻辑）
- Target 不可删除（`removeHop` 跳过 target 类型）
- 添加 SSH 跳板默认插入 Target 之前（pre-target）
- 添加 Proxy 默认插入 Target 之后（post-target / 出口链路）
- 链路可视化按 phase 着色：pre-target 蓝紫色，target 绿色，post-target 橙色

- 保存时发送 `chain` 数组到后端

### 6.3 加载兼容

编辑已有 server 时：
- 有 `chain` 数据 → 从 chain 重建 nodes（pre → target → post）
- 无 chain 数据 → 从 `proxyId` / `networkProxyId` 回退（全部为 pre-target）

## 十、已知限制

2. **ssh_test 尚未使用 chain**: 测试连接仍走旧的 4-branch 逻辑（`proxyId` + `networkProxyId`），需后续迁移。
3. **配置导出/导入**: `connection_chain` 表数据尚未纳入 config export/import，需在下一迭代补充。
