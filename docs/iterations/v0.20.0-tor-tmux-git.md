# V0.20.0 — Tor 代理 + tmux 持久会话 + Git Auto Sync

## Context

构建"数字游民"远程开发工作流：开发者从全球任意位置，通过匿名/静态住宅代理链连接远程 IDC 服务器运行 Codex 等 AI 编码工具，断开 SSH 后任务继续执行，重连后自动恢复现场，远程 AI 完成任务后自动 git push，本地自动感知并 pull 同步代码。

### 当前状态

- v0.18.0 已支持 SOCKS5/SOCKS4/HTTP/HTTPS 代理，支持代理+跳板机组合链路
- SSH 断开后远程进程终止（无持久会话支持）
- 无远程代码自动同步能力

### 设计决策（用户确认）

- Tor 方案：**外部 Tor + 自动检测**（轻量，不嵌入 arti）
- 代码同步：**Git Auto Sync**（远程自动 push → 本地自动 pull，非手动面板）
- 远程开发深度：**轻量方案**（代理链 + 持久会话 + 自动同步）
- tmux 是持久会话的关键：断开后任务继续运行，重连后 `tmux attach` 恢复

## 一、Tor 代理集成

### 1.1 方案说明

Tor 对外暴露标准 SOCKS5 接口（默认 `127.0.0.1:9050`），连接逻辑直接复用已有的 `connect_socks5()` 实现。用户也可以手动配置 SOCKS5 代理指向 Tor，但原生 Tor 类型提供更好的体验：

- 自动检测本地 Tor 服务状态
- 选择 Tor 类型时自动填充默认地址和端口
- 专属图标区分

### 1.2 Tor 自动检测

| 端口 | 来源 | 说明 |
|------|------|------|
| 9050 | 系统 Tor 服务 | `tor` daemon 默认监听端口 |
| 9150 | Tor Browser | Tor Browser Bundle 内置代理 |

检测方式：TCP connect 探测，1 秒超时。

```rust
pub async fn tor_detect() -> Result<TorStatus, String> {
    // 尝试连接 127.0.0.1:9050（系统 Tor）
    // 失败则尝试 127.0.0.1:9150（Tor Browser）
    // 返回 { running: bool, port: u16 }
}
```

跨平台兼容：TCP 探测在 macOS/Windows/Linux 行为一致，无需平台特定代码。

### 1.3 连接架构

```
Tor 代理连接:   TcpStream → Tor SOCKS5 (127.0.0.1:9050) → [Tor Network] → Exit Node → SSH Target
组合链路:       TcpStream → Tor SOCKS5 → [Tor Network] → Bastion SSH → direct-tcpip → Target SSH
```

### 1.4 Tor 局限性（重要提示）

| 特性 | 说明 |
|------|------|
| 延迟 | 300-500ms+，交互式 SSH 体验一般 |
| 出口 IP | 随时变化，无法保持静态 IP |
| 封锁 | 部分云服务商/GitHub 封锁 Tor 出口节点 |
| 适用场景 | 匿名需求 > 稳定性需求时使用 |

**推荐搭配**：Tor 作为可选匿名层，主通道使用静态住宅代理（SOCKS5/HTTP）保证稳定性。

### 1.5 UI

- ProxyTree 新增 Tor 类型，紫色洋葱形 SVG 图标
- 添加代理对话框选择 Tor 时：
  - 自动填充 Host: `127.0.0.1`, Port: `9050`
  - 触发 `tor_detect` 显示检测状态（绿色 = 运行中 / 红色 = 未检测到）
  - 不显示 TLS 选项（Tor 自带加密）

## 二、tmux 持久会话

### 2.1 核心问题

| 场景 | 无 tmux | 有 tmux |
|------|---------|---------|
| SSH 断开 | 远程进程被 SIGHUP 杀死 | 进程在 tmux session 中继续运行 |
| SSH 重连 | 丢失所有上下文 | `tmux attach` 恢复完整终端状态 |
| Codex 任务 | 任务中断，需要重新开始 | 任务继续执行，重连后看到结果 |

### 2.2 设计方案

**tmux 模式**（per-server 设置，存储在 servers 表）：

| 模式 | 行为 |
|------|------|
| `disabled` | 默认，普通 shell（当前行为不变） |
| `auto` | 检测远程 tmux → 可用则自动启用，不可用则降级为普通 shell |
| `always` | 始终启用 → 不可用时显示错误提示 |

**tmux session 命名**：`termex-{serverId 前 8 位}`（简短唯一）

### 2.3 连接流程

```
ssh_connect()          → SSH 认证
    ↓
ssh_open_shell()       → 打开 PTY shell
    ↓
ssh_exec("command -v tmux")  → 通过独立 exec channel 检测 tmux（不干扰 shell）
    ↓
if (tmuxMode !== "disabled" && tmux 可用):
    ssh_write("tmux new-session -A -s termex-{id8}\n")
    ↑ -A flag: session 存在则 attach，不存在则 create
    ↓
用户进入 tmux 环境，可以运行 Codex 等长时间任务
```

**断开重连**：
```
用户关闭 SSH 窗口
    ↓
tmux 进程继续运行在远程服务器
远程 Codex 任务继续执行
    ↓
用户重新连接同一服务器
    ↓
ssh_open_shell() → ssh_write("tmux new-session -A -s termex-{id8}\n")
    ↓
-A flag 检测到已有 session → 自动 attach
    ↓
用户看到之前的终端输出，Codex 任务结果完整保留
```

### 2.4 exec_command — 关键基础设施

SSH 协议支持在同一连接上打开多个 channel。`exec_command()` 通过独立的 exec channel 执行命令，不干扰用户的 PTY shell：

```rust
// session.rs
pub async fn exec_command(&self, command: &str) -> Result<(String, u32), SshError> {
    let channel = self.handle.channel_open_session().await?;
    channel.exec(true, command).await?;
    // 收集输出直到 EOF，返回 (stdout, exit_code)
}
```

此方法同时服务于：
1. tmux 可用性检测（`command -v tmux`）
2. Git Auto Sync 远程脚本部署（写入 `~/.termex/git-sync.sh`）

### 2.5 数据库变更

Migration v8：
```sql
ALTER TABLE servers ADD COLUMN tmux_mode TEXT DEFAULT 'disabled';
```

### 2.6 UI

ConnectModal → Authorization tab 中，Startup Command 下方新增：
```
tmux 模式  [Disabled ▼]
           ├─ Disabled   — 普通 shell
           ├─ Auto       — 自动检测并使用
           └─ Always     — 始终使用（需远程安装 tmux）
```

## 三、Git Auto Sync — 远程推送本地自动拉取

### 3.1 核心工作流

```
远程 IDC 服务器                                    本地 Termex
──────────────                                    ──────────
tmux 中运行: codex "..." && ~/.termex/git-sync.sh
    ↓ Codex 完成
git-sync.sh: git add -A && git commit && git push
    ↓ push 成功
curl http://127.0.0.1:19527/push-done             ← SSH 反向端口转发
    │                                               ↓ 本地收到通知（< 1 秒）
    │                                              [通知模式] 桌面通知 "远程 push 完成"
    │                                              [自动模式] git pull → 桌面通知 "同步完成"
    └──────────────────────────────────────────────→ 本地代码已同步，开始调试
```

**不做手动 Git 面板**——自动化是核心价值。用户不需要在 GUI 中点 Pull/Push/Commit。

### 3.2 通知架构 — SSH 反向端口转发

利用已有的 SSH 连接建立反向隧道，远程完成 push 后通过隧道实时通知本地：

```
                    SSH 连接（已存在）
本地 Termex ◄═══════════════════════════► 远程服务器
     │                                        │
     ├─ 启动本地 HTTP 监听 (127.0.0.1:0)     │
     │  随机端口，仅本地可访问                   │
     │                                        │
     └─ 建立反向转发 ─────────────────────────→ 远程 127.0.0.1:19527
                                               │
                                               ├─ git-sync.sh 执行完毕
                                               └─ curl http://127.0.0.1:19527/push-done
                                                        │
                                                        ↓ 通过 SSH 隧道传回
                                               本地 HTTP handler 收到请求
                                                        ↓
                                               触发 git pull / 桌面通知
```

端口 `19527` 仅监听在远程的 `127.0.0.1`，不对外暴露，安全无风险。

### 3.3 备用通知 — 本地 Git 轮询

SSH 断开后反向隧道不可用，退化为本地轮询：

| 通道 | 机制 | 延迟 | 适用场景 |
|------|------|------|----------|
| **主通道** | SSH 反向端口转发 + curl | < 1 秒 | SSH 在线时（实时） |
| **备用** | 本地 `git ls-remote origin HEAD` 轮询 | 30 秒 | SSH 断开后，本地仍能感知 GitHub 新 commit |

备用轮询完全在本地执行，只需网络能访问 Git 仓库（GitHub/GitLab 等），不依赖 SSH。

### 3.4 远程同步脚本 — ~/.termex/git-sync.sh

**通用脚本**，不绑定任何 AI CLI，适配 Codex/Aider/Claude Code 等任意工具：

```bash
#!/bin/bash
# ~/.termex/git-sync.sh — Termex Auto Git Sync
# 用法: codex "task" && ~/.termex/git-sync.sh [path]
#       aider --message "fix" && ~/.termex/git-sync.sh
#       任何命令 && ~/.termex/git-sync.sh

cd "${1:-.}" || exit 1

# 检查是否有变更
if git diff --quiet HEAD 2>/dev/null && [ -z "$(git ls-files --others --exclude-standard)" ]; then
    echo "[termex-sync] No changes to commit"
    exit 0
fi

# 自动 commit + push
git add -A
git commit -m "auto: $(date +%Y%m%d-%H%M%S)"
if git push; then
    echo "[termex-sync] Push successful"
    # 通知本地 Termex（反向隧道，失败静默忽略）
    curl -s -m 2 http://127.0.0.1:19527/push-done 2>/dev/null || true
else
    echo "[termex-sync] Push failed"
    exit 1
fi
```

**部署时机**：用户在 Termex 中启用 Auto Sync 设置时，自动通过 `exec_command` 写入远程。

**不在启用之前部署**——脚本只在用户明确开启功能时才写入远程服务器。

### 3.5 本地 Auto Pull

用户在 per-server 设置中配置：

| 字段 | 说明 |
|------|------|
| `git_sync_enabled` | 是否启用 Auto Sync |
| `git_sync_mode` | `notify`（默认，仅通知）/ `auto_pull`（自动 pull） |
| `git_sync_local_path` | 本地仓库路径（auto_pull 模式必填） |
| `git_sync_remote_path` | 远程仓库路径 |

**本地 pull 执行**：Rust 通过 `std::process::Command` 调用系统 `git pull`：
```rust
std::process::Command::new("git")
    .args(["pull"])
    .current_dir(&local_path)
    .output()
```

**桌面通知**：通过 Tauri 的 `notification` API 发送系统通知。

### 3.6 反向端口转发实现

利用 russh 已有的 `channel_open_direct_tcpip` / `tcpip_forward` 能力：

```rust
// 1. 本地启动 HTTP 监听（随机端口）
let listener = TcpListener::bind("127.0.0.1:0").await?;
let local_port = listener.local_addr()?.port();

// 2. 请求远程反向转发: remote 127.0.0.1:19527 → local listener
session.handle().tcpip_forward("127.0.0.1", 19527).await?;

// 3. 后台任务接收 forwarded-tcpip channel
// 当远程 curl 访问 127.0.0.1:19527 时，russh 触发 ForwardedTcpIp 回调
// 解析 HTTP 请求，若路径为 /push-done → 触发本地 git pull / 通知
```

**重连恢复**：SSH 重连后自动重建反向转发，无需用户干预。
