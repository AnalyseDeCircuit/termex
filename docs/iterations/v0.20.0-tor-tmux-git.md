# V0.20.0 — Tor 代理 + tmux 持久会话 + Git Auto Sync

## Context

构建"数字游民"远程开发工作流：开发者从全球任意位置，通过匿名/静态住宅代理链连接远程 IDC 服务器运行 Codex 等 AI 编码工具，断开 SSH 后任务继续执行，重连后自动恢复现场，远程 AI 完成任务后自动 git push，本地自动感知并 pull 同步代码。

### 当前状态

- v0.18.0 已支持 SOCKS5/SOCKS4/HTTP/HTTPS 代理，支持代理+跳板机组合链路
- SSH 断开后远程进程终止（无持久会话支持）
- 无远程代码自动同步能力
- `russh = "0.46"`，不支持反向端口转发（`tcpip_forward` / `forwarded_tcpip` 需 0.47+）
- 无桌面通知能力（`tauri-plugin-notification` 未安装）

### 设计决策（用户确认）

- Tor 方案：**外部 Tor + 自动检测**（轻量，不嵌入 arti）
- 代码同步：**Git Auto Sync**（远程自动 push → 本地自动 pull，非手动面板）
- 远程开发深度：**轻量方案**（代理链 + 持久会话 + 自动同步）
- tmux 是持久会话的关键：断开后任务继续运行，重连后 `tmux attach` 恢复

## 一、Tor 代理集成

### 1.1 方案说明

Tor 对外暴露标准 SOCKS5 接口（默认 `127.0.0.1:9050`），连接逻辑直接复用已有的 `connect_socks5()` 实现。用户也可以手动配置 SOCKS5 代理指向 Tor，但原生 Tor 类型提供更好的体验：

- 自动检测本地 Tor 服务状态
- 选择 Tor 类型时自动填充默认地址和端口
- 专属图标区分

### 1.2 Tor 自动检测

| 端口 | 来源 | 说明 |
|------|------|------|
| 9050 | 系统 Tor 服务 | `tor` daemon 默认监听端口 |
| 9150 | Tor Browser | Tor Browser Bundle 内置代理 |

检测方式：TCP connect 探测，1 秒超时。

```rust
pub async fn tor_detect() -> Result<TorStatus, String> {
    // 尝试连接 127.0.0.1:9050（系统 Tor）
    // 失败则尝试 127.0.0.1:9150（Tor Browser）
    // 返回 { running: bool, port: u16 }
}
```

跨平台兼容：TCP 探测在 macOS/Windows/Linux 行为一致，无需平台特定代码。

### 1.3 连接架构

```
Tor 代理连接:   TcpStream → Tor SOCKS5 (127.0.0.1:9050) → [Tor Network] → Exit Node → SSH Target
组合链路:       TcpStream → Tor SOCKS5 → [Tor Network] → Bastion SSH → direct-tcpip → Target SSH
```

### 1.4 Tor 局限性（重要提示）

| 特性 | 说明 |
|------|------|
| 延迟 | 300-500ms+，交互式 SSH 体验一般 |
| 出口 IP | 随时变化，无法保持静态 IP |
| 封锁 | 部分云服务商/GitHub 封锁 Tor 出口节点 |
| 适用场景 | 匿名需求 > 稳定性需求时使用 |

**推荐搭配**：Tor 作为可选匿名层，主通道使用静态住宅代理（SOCKS5/HTTP）保证稳定性。

### 1.5 数据库兼容性

proxies 表的 `proxy_type` 字段为 TEXT 类型，可直接存储 `"tor"` 字符串，**无需 migration**。仅需：
- Rust `ProxyType` enum 添加 `Tor` variant
- 前端 `ProxyType` 联合类型添加 `"tor"`
- 连接时 `Tor` 委托给 `connect_socks5()`（复用已有实现）

### 1.6 测试连接

选择 Tor 类型后点击 "Test Connection"：
1. TCP connect 探测目标端口（9050/9150）—— 验证 Tor 服务是否可达
2. 执行 SOCKS5 握手（不建立 SSH）—— 验证 SOCKS5 协议正常
3. 返回 `{ reachable: bool, socks5_ok: bool, exit_ip: Option<String> }`

`exit_ip` 为可选项：如能通过 Tor SOCKS5 请求外部 IP 检测服务（如 `https://check.torproject.org/api/ip`），则返回出口 IP 供用户确认匿名性。

### 1.7 UI

- ProxyTree 新增 Tor 类型，紫色洋葱形 SVG 图标
- 添加代理对话框选择 Tor 时：
  - 自动填充 Host: `127.0.0.1`, Port: `9050`
  - 触发 `tor_detect` 显示检测状态（绿色 = 运行中 / 红色 = 未检测到）
  - 不显示 TLS 选项（Tor 自带加密）

## 二、tmux 持久会话

### 2.1 核心问题

| 场景 | 无 tmux | 有 tmux |
|------|---------|---------|
| SSH 断开 | 远程进程被 SIGHUP 杀死 | 进程在 tmux session 中继续运行 |
| SSH 重连 | 丢失所有上下文 | `tmux attach` 恢复完整终端状态 |
| Codex 任务 | 任务中断，需要重新开始 | 任务继续执行，重连后看到结果 |

### 2.2 设计方案

**tmux 模式**（per-server 设置，存储在 servers 表）：

| 模式 | 行为 |
|------|------|
| `disabled` | 默认，普通 shell（当前行为不变） |
| `auto` | 检测远程 tmux → 可用则自动启用，不可用则降级为普通 shell |
| `always` | 始终启用 → 不可用时显示错误提示 |

**tmux session 命名**：`termex-{serverId 前 8 位}-{sessionId 前 4 位}`

- 同一服务器打开多个 tab 时，每个 tab 拥有独立的 tmux session（如 `termex-a1b2c3d4-f7e2`、`termex-a1b2c3d4-9c1a`）
- 使用 `sessionId`（UUID）前 4 位作为后缀，保证唯一性且无需修改 sessionStore 结构
- sessionStore 当前 `tabKey` 格式为 `tab-{timestamp}-{randomStr}`，不适合做 tmux session 名，故采用 sessionId

### 2.3 tmux session 生命周期

| 事件 | 行为 | 说明 |
|------|------|------|
| 用户关闭 tab | **detach**（默认） | 远程 tmux session 继续运行，下次连接可 attach 恢复 |
| 用户关闭 tab | **kill**（可配置） | 立即销毁远程 tmux session，释放资源 |
| SSH 断线（网络中断） | 自动 detach | tmux 进程不受影响，重连后 attach |
| 服务器重启 | tmux session 丢失 | 下次连接自动创建新 session |

**per-server 配置**：`tmux_close_action`：`detach`（默认）| `kill`

关闭 tab 时：
- `detach`：仅断开 SSH，tmux session 保留（远程任务继续运行）
- `kill`：通过 exec_command 执行 `tmux kill-session -t {session_name}` 后断开 SSH

**kill-session 执行时机**（Rust 端 `ssh_disconnect` 命令中）：

```
ssh_disconnect() 现有流程：
  1. SFTP cleanup（sftp_sessions.remove）
  2. Session remove（sessions.remove）
  3. session.disconnect()（关闭 SSH 连接）
  4. Proxy ref cleanup

kill-session 插入位置：在步骤 2 之后、步骤 3 之前
  → session.exec_command("tmux kill-session -t {name}")
  → 此时 SSH 连接仍然存活，exec_command 可正常执行
  → 执行完毕后再 session.disconnect()
```

`tmux_close_action` 和 `session_name` 需从前端传入 `ssh_disconnect` 命令参数，
或由 Rust 端从 session 元数据中读取（需在 session 中存储 tmux 状态）。

### 2.4 startup_cmd 与 tmux 的交互

服务器已有 `startup_cmd` 字段（SSH 连接后自动执行的初始化命令，如 `cd /project && source venv/bin/activate`）。

**策略：条件执行** — 仅在**新建** tmux session 时执行 startup_cmd，**重连 attach** 时跳过：

```
1. exec_command("tmux has-session -t {session_name} 2>/dev/null")
2. 如果 session 不存在（exit_code != 0）:
   → ssh_write("tmux new-session -s {session_name}\n")
   → ssh_write("{startup_cmd}\n")   ← 首次，初始化环境
3. 如果 session 已存在（exit_code == 0）:
   → ssh_write("tmux attach -t {session_name}\n")
   → 不执行 startup_cmd            ← 环境已在，保留现场
```

**原因**：
- 首次连接：用户需要 cd 到项目目录、激活虚拟环境等初始化操作
- 断线重连：tmux 保留了完整的 shell 状态（工作目录、环境变量、运行中的进程），再次执行 startup_cmd 会干扰现场

当 tmux 为 `disabled` 时，startup_cmd 行为不变（直接在普通 shell 中执行）。

### 2.5 tmux 状态指示

StatusBar 新增 tmux 指示器（位于连接状态文本右侧）：

| 状态 | 显示 | 说明 |
|------|------|------|
| tmux disabled | 不显示 | 普通 shell 模式 |
| tmux 检测中 | 灰色 `tmux` 文本 | auto 模式检测阶段 |
| tmux 已进入 | 绿色 `tmux` 标签 | 当前在 tmux session 中 |
| tmux 不可用 | 橙色 `tmux ✗` | auto 模式降级 / always 模式失败 |

### 2.6 连接流程

```
ssh_connect()          → SSH 认证
    ↓
ssh_open_shell()       → 打开 PTY shell
    ↓
ssh_exec("command -v tmux")  → 通过独立 exec channel 检测 tmux（不干扰 shell）
    ↓
if (tmuxMode !== "disabled" && tmux 可用):
    ssh_exec("tmux has-session -t {session_name} 2>/dev/null")
    ↓
    exit_code != 0 (无已有 session):
        ssh_write("tmux new-session -s {session_name}\n")
        ssh_write("{startup_cmd}\n")     ← 首次，初始化环境
    exit_code == 0 (已有 session):
        ssh_write("tmux attach -t {session_name}\n")
                                          ← 跳过 startup_cmd，保留现场
    ↓
用户进入 tmux 环境，可以运行 Codex 等长时间任务

if (tmuxMode === "always" && tmux 不可用):
    → 前端提示错误：远程未安装 tmux

if (tmuxMode === "auto" && tmux 不可用):
    → 降级为普通 shell，执行 startup_cmd（如有）
```

**断开重连**：
```
用户关闭 SSH tab（tmux_close_action = detach）
    ↓
SSH 断开，tmux session 自动 detach
远程 Codex 任务继续在 tmux 中执行
    ↓
用户重新连接同一服务器（新 tab）
    ↓
ssh_open_shell()
    ↓
ssh_exec("tmux has-session -t termex-{id8}-{sid4}")  → exit_code == 0
    ↓
ssh_write("tmux attach -t termex-{id8}-{sid4}\n")
    ↓
用户看到之前的终端输出，Codex 任务结果完整保留
（startup_cmd 不再执行，保持现场）
```

**错误降级**：
```
tmuxMode === "auto" && tmux 可用:
    ssh_exec("tmux has-session ...") 成功但 tmux new-session 失败（权限等）
    ↓
    降级为普通 shell + startup_cmd
    前端 StatusBar 显示橙色 "tmux ✗"
    ElMessage.warning("tmux session failed, falling back to normal shell")
```

### 2.7 exec_command — 关键基础设施

SSH 协议支持在同一连接上打开多个 channel。`exec_command()` 通过独立的 exec channel 执行命令，不干扰用户的 PTY shell：

```rust
// session.rs
pub async fn exec_command(&self, command: &str) -> Result<(String, u32), SshError> {
    let channel = self.handle.channel_open_session().await?;
    channel.exec(true, command).await?;
    // 收集输出直到 EOF，返回 (stdout, exit_code)
}
```

此方法同时服务于：
1. tmux 可用性检测（`command -v tmux`）
2. Git Auto Sync 远程脚本部署（写入 `~/.termex/git-sync.sh`）

### 2.8 数据库变更

Migration v8：
```sql

ALTER TABLE servers ADD COLUMN tmux_close_action TEXT DEFAULT 'detach';
```

### 2.9 UI

ConnectModal → Authorization tab 中，Startup Command 下方新增：
```

                  ├─ Disabled   — 普通 shell
                  ├─ Auto       — 自动检测并使用

```

## 三、Git Auto Sync — 远程推送本地自动拉取

### 3.1 核心工作流

```
远程 IDC 服务器                                    本地 Termex
──────────────                                    ──────────
tmux 中运行: codex "..." && ~/.termex/git-sync.sh
    ↓ Codex 完成
git-sync.sh: git add -A && git commit && git push
    ↓ push 成功
curl http://127.0.0.1:19527/push-done             ← SSH 反向端口转发
    │                                               ↓ 本地收到通知（< 1 秒）
    │                                              [通知模式] 桌面通知 "远程 push 完成"
    │                                              [自动模式] git pull → 桌面通知 "同步完成"
    └──────────────────────────────────────────────→ 本地代码已同步，开始调试
```

**不做手动 Git 面板**——自动化是核心价值。用户不需要在 GUI 中点 Pull/Push/Commit。

### 3.2 通知架构 — SSH 反向端口转发

利用已有的 SSH 连接建立反向隧道，远程完成 push 后通过隧道实时通知本地：

```
                    SSH 连接（已存在）
本地 Termex ◄═══════════════════════════► 远程服务器
     │                                        │
     ├─ 启动本地 HTTP 监听 (127.0.0.1:0)     │
     │  随机端口，仅本地可访问                   │
     │                                        │
     └─ 建立反向转发 ─────────────────────────→ 远程 127.0.0.1:19527
                                               │
                                               ├─ git-sync.sh 执行完毕
                                               └─ curl http://127.0.0.1:19527/push-done
                                                        │
                                                        ↓ 通过 SSH 隧道传回
                                               本地 HTTP handler 收到请求
                                                        ↓
                                               触发 git pull / 桌面通知
```

端口 `19527` 仅监听在远程的 `127.0.0.1`，不对外暴露，安全无风险。

### 3.3 备用通知 — 本地 Git 轮询

SSH 断开后反向隧道不可用，退化为本地轮询：

| 通道 | 机制 | 延迟 | 适用场景 |
|------|------|------|----------|
| **主通道** | SSH 反向端口转发 + curl | < 1 秒 | SSH 在线时（实时） |
| **备用** | 本地 `git ls-remote origin HEAD` 轮询 | 60 秒（默认） | SSH 断开后，本地仍能感知 GitHub 新 commit |

备用轮询完全在本地执行，只需网络能访问 Git 仓库（GitHub/GitLab 等），不依赖 SSH。

**轮询频率**：默认 60 秒，可在 Settings → General 中调整（30s / 60s / 120s / 300s）。较低频率避免触发 GitHub API rate limit（未认证: 60次/小时，认证: 5000次/小时）。

**轮询触发条件**：仅在 SSH 断开 + git_sync_enabled + git_sync_local_path 存在时启动轮询。SSH 重连后自动停止轮询，切回反向转发主通道。

### 3.4 远程同步脚本 — ~/.termex/git-sync.sh

**通用脚本**，不绑定任何 AI CLI，适配 Codex/Aider/Claude Code 等任意工具：

```bash
#!/bin/bash
# ~/.termex/git-sync.sh — Termex Auto Git Sync
# 用法: codex "task" && ~/.termex/git-sync.sh [path]
#       aider --message "fix" && ~/.termex/git-sync.sh
#       任何命令 && ~/.termex/git-sync.sh

cd "${1:-.}" || exit 1

# 检查是否有变更
if git diff --quiet HEAD 2>/dev/null && [ -z "$(git ls-files --others --exclude-standard)" ]; then
    echo "[termex-sync] No changes to commit"
    exit 0
fi

# 自动 commit + push
git add -A
git commit -m "auto: $(date +%Y%m%d-%H%M%S)"
SYNC_PORT=$(cat ~/.termex/sync-port 2>/dev/null || echo 19527)
if git push; then
    echo "[termex-sync] Push successful"
    # 通知本地 Termex（反向隧道，失败静默忽略）
    curl -s -m 2 "http://127.0.0.1:${SYNC_PORT}/push-done" 2>/dev/null || true
else
    echo "[termex-sync] Push failed"
    exit 1
fi
```

#### 3.4.1 脚本部署

**部署时机**：用户在 Termex 中启用 Auto Sync 且 SSH 连接成功时，自动通过 `exec_command` 部署。

**不在启用之前部署**——脚本只在用户明确开启功能时才写入远程服务器。

**部署流程**（通过 exec_command 链式执行）：
```bash
mkdir -p ~/.termex && \
cat > ~/.termex/git-sync.sh << 'TERMEX_SYNC_EOF'
#!/bin/bash
# ... 脚本内容 ...
TERMEX_SYNC_EOF
chmod +x ~/.termex/git-sync.sh && \
echo {sync_port} > ~/.termex/sync-port
```

**脚本更新策略**：
- 每次 SSH 连接成功且 git_sync_enabled 时，对比远程脚本版本
- 版本号写入 `~/.termex/sync-version`（如 `0.20.0`）
- 本地 Termex 版本 > 远程版本时，重新部署脚本
- 版本一致时跳过部署（避免每次连接的 exec 开销）

#### 3.4.2 安全注意事项

`git add -A` 会提交工作目录下所有文件，包括可能的敏感文件。**安全依赖于远程仓库的 `.gitignore` 配置**。

Termex 不对此做额外限制（这是用户自己仓库的 `.gitignore` 职责），但在 Sync 标签页中显示提示文本：

> 请确保远程仓库的 `.gitignore` 已正确配置，排除 `.env`、密钥文件等敏感数据。

### 3.5 本地 Auto Pull

用户在 per-server 设置中配置：

| 字段 | 说明 |
|------|------|
| `git_sync_enabled` | 是否启用 Auto Sync |
| `git_sync_mode` | `notify`（默认，仅通知）/ `auto_pull`（自动 pull） |
| `git_sync_local_path` | 本地仓库路径（auto_pull 模式必填） |
| `git_sync_remote_path` | 远程仓库路径 |

**本地 pull 执行**：Rust 通过 `std::process::Command` 调用系统 `git pull`：
```rust
std::process::Command::new("git")
    .args(["pull"])
    .current_dir(&local_path)
    .output()
```

**桌面通知**：通过 Tauri 的 `notification` API 发送系统通知。

### 3.6 反向端口转发实现

当前代码库仅支持本地端口转发（`-L`），**反向转发（`-R`）需新建**。

#### 前置条件：russh 升级

当前 `russh = "0.46"` **不支持** `tcpip_forward()` 和 `forwarded_tcpip()` 回调。
必须升级到 **russh 0.47+** 才能使用反向端口转发。

**升级步骤**（作为 Step 1 执行）：
1. `Cargo.toml`：`russh = "0.47"` + `russh-keys = "0.47"`
2. 检查 breaking changes（0.46 → 0.47 API 变更）
3. 在独立分支完成全量 SSH 功能回归测试（连接、认证、SFTP、端口转发、exec_command）
4. 确认通过后合并

#### 3.6.2 建立反向转发流程

```rust
// SSH 连接成功 + git_sync_enabled 时执行：

// 1. 请求远程反向转发
let port = calculate_sync_port(server_id);  // 见 3.6.3
session.handle().tcpip_forward("127.0.0.1", port).await?;

// 2. 注册到 ReverseForwardRegistry
// 当 Handler::forwarded_tcpip 收到数据时，解析 HTTP 请求

// 3. HTTP 请求处理（极简解析，无需完整 HTTP 库）
fn parse_sync_request(data: &[u8]) -> Option<SyncAction> {
    let request = String::from_utf8_lossy(data);
    if request.contains("GET /push-done") {
        Some(SyncAction::PushDone)
    } else {
        None
    }
}

// 4. 触发本地动作
match action {
    SyncAction::PushDone => {
        app_handle.emit("git-sync://push-done", &server_id)?;
        // 前端收到事件后根据 mode 执行：
        // - notify: 桌面通知
        // - auto_pull: git pull → 通知
    }
}
```

**不引入 HTTP 库**（如 `tiny_http`/`hyper`）— 仅需解析 `GET /push-done`，手写字符串匹配即可，零依赖。

#### 3.6.3 远程端口冲突策略

固定端口 19527 在同服务器多连接时会冲突。解决方案：

```rust
/// 基于 server_id 计算唯一端口，范围 19500-19999
fn calculate_sync_port(server_id: &str) -> u32 {
    let hash = server_id.as_bytes().iter().fold(0u32, |acc, &b| acc.wrapping_add(b as u32));
    19500 + (hash % 500)
}
```

- 每个服务器 ID 映射到固定端口（19500-19999 范围）
- 同一服务器多 tab 共享同一端口（反向转发由 SSH 连接绑定，不同 tab 的 SSH 连接独立）
- 远程脚本 `git-sync.sh` 从 `~/.termex/sync-port` 文件读取端口号（部署时写入）

更新后的 `git-sync.sh`（见 3.4）中 curl 目标：
```bash
SYNC_PORT=$(cat ~/.termex/sync-port 2>/dev/null || echo 19527)
curl -s -m 2 "http://127.0.0.1:${SYNC_PORT}/push-done" 2>/dev/null || true
```

#### 3.6.4 重连恢复

SSH 重连后自动重建反向转发，无需用户干预：
1. `ssh_connect` 成功
2. 检查 `server.git_sync_enabled`
3. 重新 `tcpip_forward` + 注册 handler
4. 重新部署 `~/.termex/sync-port`（端口不变，但确保文件存在）

## 四、错误处理与降级策略

### 4.1 Tor 代理

| 场景 | 处理 |
|------|------|
| Tor 服务未运行 | `tor_detect` 返回 `running: false`，UI 红色状态提示，不阻止保存（用户可能稍后启动 Tor） |
| SOCKS5 握手失败 | 复用已有 proxy 错误处理，`SshError::ProxyFailed` → 前端提示 |
| Tor 出口被目标封锁 | SSH 连接超时，正常超时处理 |

### 4.2 tmux

| 场景 | 处理 |
|------|------|
| `command -v tmux` 失败（未安装） | `auto` → 降级普通 shell + StatusBar 橙色提示；`always` → ElMessage.error + 仍开普通 shell |
| `tmux new-session` 失败（权限等） | 降级普通 shell + 执行 startup_cmd + StatusBar 橙色提示 |
| `tmux attach` 失败（session 被外部 kill） | 自动 fallback 为 `tmux new-session`（创建新 session） + 执行 startup_cmd |
| tmux 版本过旧（不支持 `-A` flag） | 使用 `has-session` + `new-session` / `attach` 分支逻辑（已在 2.6 流程中采用，不依赖 `-A`） |

### 4.3 Git Auto Sync

| 场景 | 处理 |
|------|------|
| `tcpip_forward` 失败（端口被占用） | 尝试 +1 端口重试（最多 3 次），全部失败则禁用反向转发，退化为轮询模式 |
| 远程 `mkdir -p ~/.termex` 失败（权限） | ElMessage.error 提示，git_sync 标记为未激活 |
| 本地 `git pull` 冲突 | 桌面通知 "同步失败：合并冲突，请手动解决" + StatusBar 红色叉 |
| 本地 `git pull` 路径不存在 | 桌面通知 "本地仓库路径不存在" + StatusBar 红色叉 |
| 远程脚本执行但 `git push` 失败 | 脚本输出 `[termex-sync] Push failed`，不触发 curl（本地无感知，用户在终端看到错误） |
| SSH 断开后轮询 `git ls-remote` 失败 | 静默忽略，下次轮询周期重试 |

### 4.4 功能交互矩阵

| 组合 | 行为 | 注意事项 |
|------|------|----------|
| Tor + tmux | 正常工作 | Tor 延迟 300-500ms，tmux 操作体感略慢但功能不受影响 |
| Tor + Git Sync | 正常工作 | 反向转发通过 Tor SSH 隧道建立，curl 通知走隧道回传，延迟增加但 < 2 秒 |
| tmux + Git Sync | **推荐组合** | tmux 保持 Codex 运行，git-sync.sh 在 tmux 中执行后通知本地 |
| Tor + tmux + Git Sync | 全部正常 | "数字游民"完整工作流 |
